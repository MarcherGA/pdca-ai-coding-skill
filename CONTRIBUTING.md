# Contributing to PDCA AI Coding Skill

Thank you for your interest in contributing! This skill is designed to evolve through community use and feedback.

## Ways to Contribute

### 1. Share Your Experience
- Use the skill for 2-4 weeks
- Document what works and what doesn't
- Share your retrospectives and improvements

### 2. Report Issues
Use GitHub Issues for:
- Bugs in scripts or prompts
- Unclear documentation
- Missing features
- Ideas for improvement

**Issue Template:**
```markdown
**Type:** [Bug/Enhancement/Question]
**Component:** [Prompts/Scripts/Documentation]

**Description:**
Clear description of the issue or suggestion

**Steps to Reproduce (for bugs):**
1. 
2. 
3. 

**Expected Behavior:**

**Actual Behavior:**

**Context:**
- Your use case
- How you're using the skill
- Any customizations made
```

### 3. Submit Improvements

#### For Prompt Refinements
1. Use the skill extensively
2. Conduct retrospectives
3. Identify specific prompt improvements
4. Test your changes
5. Submit PR with:
   - What changed
   - Why it improves the prompt
   - Evidence from your sessions

#### For Script Enhancements
1. Fork the repository
2. Create a feature branch: `git checkout -b feature/your-feature`
3. Make your changes with tests
4. Ensure code quality: `python -m pylint scripts/*.py`
5. Update documentation
6. Submit PR

#### For Documentation
- Fix typos, clarify instructions
- Add examples from real usage
- Improve quick start guide
- Add language-specific guides

## Contribution Guidelines

### Code Quality
- Python scripts should follow PEP 8
- Add docstrings to functions
- Include type hints where helpful
- Keep scripts simple and focused

### Prompt Quality
- Test prompts through multiple sessions
- Ensure they work with Claude Code/Cline
- Keep prompts concise (progressive disclosure)
- Maintain article's core principles

### Documentation Standards
- Clear, concise language
- Real examples over abstract descriptions
- Markdown formatting
- Keep READMEs under 300 lines

## Pull Request Process

1. **Before Starting**
   - Check existing issues/PRs
   - Discuss major changes first
   - Ensure it aligns with PDCA principles

2. **PR Description Should Include:**
   - What problem does it solve?
   - How did you test it?
   - Any breaking changes?
   - Links to related issues

3. **Review Process**
   - Maintainers will review within 1 week
   - May request changes or clarifications
   - Be open to feedback

4. **After Merge**
   - Changes will be included in next release
   - You'll be credited in release notes

## Community Guidelines

### Be Respectful
- Constructive feedback only
- Assume good intentions
- Help others learn

### Be Collaborative
- Share your learnings
- Build on others' work
- Credit contributions

### Be Practical
- Focus on real-world use
- Evidence-based improvements
- Maintain simplicity

## Recognition

Contributors will be:

## Code of Conduct

### Our Standards

**Positive behaviors:**
- Using welcoming and inclusive language
- Respecting differing viewpoints
- Accepting constructive criticism
- Focusing on what's best for the community

**Unacceptable behaviors:**
- Trolling, insulting, or derogatory comments
- Public or private harassment
- Publishing others' private information
- Other unethical or unprofessional conduct

### Enforcement

Violations can be reported to repository maintainers. All complaints will be reviewed and investigated promptly and fairly.
